# Plan

## 铁律

1. **不用计划掩盖未解决的设计问题**：方案有未决分歧，先转 `design` 或 `rfc`，不要用步骤把问题埋进去。
2. **每步必须有可观察验证**：没有验证命令或检查点的步骤，重写或删除，不妥协。
3. **超过 12 步必须拆阶段**：扁平长 list 是反模式；阶段之间设置同步点，每阶段结束可展示价值。

## 何时不用我

- **任务小于 30 分钟且改动可逆** — 直接做，不用走流程（GSD 原则：能做就别规划）。
- **技术方案还未稳定** — 转 `design` 或 `rfc`，方案稳了再回来。
- **只是一次性脚本或实验** — 直接开写，失败了扔掉，用 `diagnose` 排查问题即可。

## 何时使用

- 技术栈、模块、数据流已定义，需要把方案落成可执行步骤。
- 用户说「怎么实现？」「拆一下步骤」「列个计划」「给我一个 implementation plan」。
- 工作量超过 1 天，或涉及多人协作，需要明确任务边界和顺序。
- 准备移交给 `tdd` 前的最后一步结构化。

## 垂直切片 vs 水平层

**垂直切片**（正确）：每一片端到端能跑、能验证、能展示价值。例如「实现用户注册」= API + 存储 + 前端表单一起做完，这片功能可独立演示。

**水平层**（反模式）：「先全做后端，再全做前端，最后全做测试」。中间状态不可验证，集成 bug 最后才暴露，风险集中在末期。

按垂直切片拆，每片结束 = 一个可演示的工作增量。

需要例子或步骤模板时读 [`references/vertical-slice-examples.md`](references/vertical-slice-examples.md)。

## 流程

1. **读已确认的方案**：找 design doc、rfc、ADR，确认技术栈 / 模块 / 数据流都已锁定。
2. **识别垂直切片**：每片 = 端到端一条功能路径，不按层拆，按用户可感知的增量拆。
3. **排步骤**：每步写清楚依赖（依赖哪步完成）和并行机会（哪几步可以同时跑）。
4. **写验证**：每步配验证命令（`npm test`、`curl`、`assert`）或可观察检查（界面出现 X、日志打印 Y）。
5. **控制粒度**：每步任务量小到 fresh context 能独立完成，超过半天的步骤继续拆。
6. **标关键路径**：找出哪几步串行且在关键路径上，并行机会明确标注。
7. **识别风险**：每个主要步骤问「这里最可能卡住的是什么」，写进风险表。
8. **写第一刀**：从计划里找现在就能开始的最小动作，降低启动摩擦。
9. **完成后按风险选择**：步骤依赖仍不清楚 → `grill --align`；计划已成形且风险高 → `grill --redteam`。
10. **计划确认后移交 `tdd`**：让 `tdd` 按步骤写测试先行。
11. **需要复用时保存计划**：超过 1 天、多人协作或跨 session 的计划，写到 `docs/plans/<YYYY-MM-DD>-<slug>.md`；短计划只在回复中输出。

<一句话，说清楚完成后用户能做什么>

## 文件 / 区域

- `<...>`

|---|---|---|---|---|
| 1 | <动词 + 具体内容> | - | 可与 2 并行 | `<命令>` 或「出现 X」 |

## 关键路径
- 步骤 1 → 3 → 5 是关键路径（不可并行，决定整体时间）
- 步骤 2、4 可与关键路径并行执行

<现在可以开始的最小动作，不需要等任何前置条件完成>

## 停止条件

- **目标不清** — 转 `clarify`，问清楚再回来。
- **技术栈 / 模块 / 数据流未定** — 转 `design` 或 `rfc`，方案未稳不出计划。
- **选型未决** — 转 `decide`，先做决策再拆步骤。
- **步骤无验证** — 重写或删除，不接受「应该能跑」「看起来对」等弱证据（superpower 原则）。
- **超过 12 步** — 强制拆阶段，不接受超长扁平 list。
