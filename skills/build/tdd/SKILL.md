# TDD

## 铁律

1. 没有失败测试，就没有生产代码。没有例外，没有"先写实现再补测试"。
2. 测试走 public interface，不测 private 方法和内部实现。测试是行为合约，不是结构快照。
3. 写不出测试 = 没想清楚问题。这是 superpower 核心信念：卡在测试上是信号，去 `clarify` 或 `design`，不是跳过 TDD。

## 何时不用我

- **一次性原型**：用完即弃、不会进生产的探索代码。
- **纯配置**：YAML / JSON / 环境变量，没有逻辑分支可测。
- **生成代码**：codegen 输出，逻辑在模板层而非生成物层。
- **没有可行测试接缝**：紧耦合遗留代码无法注入依赖 — 先转 `refactor` 建 seam，再回来。
- **用户明确说不要 TDD**：GSD 逃生口，记录理由，继续交付。

## 何时使用

- 新功能：从 acceptance 层往内推，行为先于实现。
- 可测试的 bugfix：先写复现 bug 的红测试，再修代码让它变绿。
- 行为变更：先更新测试描述期望行为，再改实现。
- 保持行为不变的重构：确认全绿后重构，测试套件是安全网。

## 流程

### 红绿重构 7 步

1. **选一个行为（最小可测）** — 不是一个功能，是一个具体的输入输出断言。
2. **写一个失败测试** — 只写一个。测试名用"given_when_then"或"should_xxx"，描述行为。
3. **运行，确认失败原因正确** — 不是 setup 错误、不是 typo、不是依赖未引入。失败原因必须是"被测行为不存在"。
4. **写最小实现** — 可以硬编码，可以很丑，只要让测试通过。不过度实现。
5. **运行，确认通过** — 全部已有测试必须同时通过，不只是新测试。
6. **只在绿态重构** — 消除重复，提升可读性，不改行为。每次重构后立即重跑。
7. **小提交（git 项目）** — 绿态 + 重构完 = 一个提交。提交信息描述行为，不描述实现。

### outside-in vs inside-out

| 模式 | 方向 | 适合场景 | 优点 |
|---|---|---|---|
| outside-in | acceptance → unit | 新功能，用户行为明确 | 少 mock、行为驱动、不过度设计 |
| inside-out | 核心算法 → 外层 | 算法 / 数据结构密集任务 | 算法正确性先验证，再组装 |

**默认用 outside-in**。理由：从用户可见行为出发，mock 需求自然浮现而非提前猜测，避免过度设计内部结构。

## 好测试 vs 坏测试

| 好测试 | 坏测试 |
|---|---|
| 走 public interface | 直接调用 private method |
| 测行为（输入→输出） | 测实现（内部调用次数） |
| 能承受内部重构不改测试 | 重构一改就红 |
| 走真实代码路径 | 默认 mock 所有内部协作者 |
| 一次写一个 | 一口气写一堆再实现 |
| 断言可观测行为 | 只断言结构或返回类型 |

## 停止条件

- **实现前测试就通过** — 测试写错了，重写测试，重新走红。
- **测试因 setup / typo 报错** — 不算"红"，修测试，继续等真正的红。
- **没有 seam，无法注入依赖** — 停止，转 `refactor` 先建 seam，再回来。
- **卡在"写不出测试"** — 停止，转 `clarify` 或 `design` 想清楚问题边界。
- **完成后** — 必须走 `verify` gate，不接受"应该通过"的弱证据。
