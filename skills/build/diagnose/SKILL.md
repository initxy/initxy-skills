# Diagnose

## 铁律

1. **没有复现反馈环，就没有修复。** 凭感觉改代码不是调试，是赌博。
2. **一次只改一个变量。** 同时改多处等于消除了所有信号。
3. **试了 3 次修复仍未中根因 — 停止，重新调查。** 退回假设列表；缺信息则要日志/产物/权限。

## 何时不用我

- **bug 显然且修复 < 5 分钟** — 可直接改，但仍要保留一个最小复现/验证命令；没有验证命令时输出 `未验证`。
- **是设计问题，不是 bug** — 转 `design` 或 `refactor`。
- **测试本身写错了** — 修测试不诊断系统；属于 `tdd` 范畴。

## 何时使用

- bug / 报错 / 未捕获异常 / crash / CI 失败
- 测试失败但不知道根因
- 性能回归（指标昨天还正常）
- "昨天还好好的，今天突然不行了"
- 输出不符合预期，根因不明

## 流程

**1. 写清期望和实际** — 各一句话；含糊的问题陈述导致含糊的假设。

**2. 建最快反馈环**（执行 < 30 秒为佳）
失败测试 / shell 命令 / curl / 浏览器脚本 / trace replay / 最小 harness。

**3. 复现并记录复现率（n/m）**
100% 复现 → 确定性 bug；偶发 → 可能竞态/时序；无法复现 → 进入停止条件。

**4. 列 3-5 个可证伪假设**（见下节）
每个假设必须有可操作的验证方法，能被一个具体实验推翻。

**5. 一次只改一个变量验证**
按"最可能 + 最易验证"排序，逐一验证，记录输入/输出/结论。

**6. 修复确认过的根因**
只改根因处，不顺手清理无关代码（改多了 = 引入新变量）。

**7. 加回归覆盖**
为这个 bug 加一个最小失败测试（先红后绿）。无法加时必须说明原因。

**8. 重新跑原始复现** — 用第 2 步完全相同的命令验证修复，输出摘要写进报告，再调用 `verify`。

## 假设要可证伪

借鉴 superpower 演绎法：假设必须能被具体实验推翻（Sherlock Holmes 风格）。
卡住时读 [`references/root-cause-patterns.md`](references/root-cause-patterns.md)。

| 类型 | 示例 |
|------|------|
| 坏（不可证伪） | "是数据库慢导致的。" — 无预测，无验证方法。 |
| 好（可证伪） | "若是 DB 慢，mock 掉 DB 后延迟应立刻 < 10ms；否则瓶颈不在 DB。" |

写假设时强制填写：原因 / 预测 / 验证方法 / 实际结果。

## Bisect

何时用：知道"某提交之前是好的"但不知道哪次引入回归，且范围 > 20 个提交。

```bash
git bisect start
git bisect bad                   # 当前是坏的
git bisect good <good-commit>    # 上次已知正常
git bisect run <your-test-cmd>   # 全自动模式
git bisect reset
```

配置/数据二分：用 feature flag 或最小失败数据集逐步减半，原理相同。

## 输出

```markdown
## 现象
- 期望：<一句话>
- 实际：<一句话>

## 复现命令
<command>  →  复现率 <n/m>

## 假设表
| # | 原因 | 预测 | 结果 |
|---|------|------|------|

## 根因
<确认的原因>

## 修复
- `path/to/file`：<改了什么，为什么>

## 验证
<原始复现命令的新鲜输出摘要>
```

## 停止条件

- **无法复现** — 要求提供日志、产物、环境权限或临时打点后重跑；不猜测。
- **3 次修复未中根因** — 停止修改，重回第 4 步补充假设。
- **Debug log** — 临时日志必须标记 `[DEBUG-<id>]`，诊断完成后全部删除，不得合入主干。
- **清理 debug log** — 可运行 `scripts/find-debug-logs.sh <root>` 扫描遗留日志。
- **修复后必须调用 `verify`** — 不接受"应该好了"等弱证据；`verify` 是闸门。
- **确认是设计问题** — 转 `design` 或 `refactor`，终止诊断。
