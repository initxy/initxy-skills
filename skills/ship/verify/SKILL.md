# Verify

## 铁律

1. 没有新鲜验证证据，不能声称完成。没有例外，没有"应该没问题"。
2. 新鲜 = 本次对话中现场跑出来的输出。缓存结果、子 agent 报告、回忆输出——都不算。
3. 你负责跑命令、读输出，汇报实际结果，不是预测结果。

## 何时不用我

- **纯讨论 / 探索 / 设计阶段**：还没有产物，不需要验证。
- **用户明确说"先不验证，我自己跑"**：GSD 逃生口，记录未验证状态，继续交付。
- **30 分钟内可完全撤销的实验**：验证成本高于风险时，说明理由跳过。

## 何时使用

- 准备说 done / fixed / passing / ready / shipped 之前。
- 提交、PR、release、deploy 之前（`ship` 会调用我）。
- `tdd` 绿态后正式结案前。
- `diagnose` 修复完成，准备关闭 bug 之前。
- 用户问"测了吗？""跑过了吗？""确认可以 merge 吗？"

## 新鲜的定义

同一 session 内满足以下**全部**条件才算新鲜：

- 输出产生于本次对话（≤ 5 分钟前）。
- 产物文件在那之后没有被修改过。
- 不是子 agent 的汇报——必须是你自己读到的原始输出。

超过 5 分钟、或文件有改动——必须重新运行。

## 验证门（4 步）

1. **找命令**：确定能证明声明成立的命令或可观察行为（测试命令、build 命令、curl、浏览器流程）。
2. **现场运行**：不复用旧输出，重新执行。
3. **读完整输出 + 退出码**：不跳过错误行，不只看最后一行。
4. **汇报实际结果**：不预测，不解释，直接给证据。

## 声明 vs 证据对照表

| 声明 | 必须提供的证据 |
|---|---|
| tests pass | 测试命令原始输出，0 failures，exit 0 |
| build works | build 命令，exit 0，无 error 行 |
| bug fixed | 原始复现命令不再失败 |
| regression covered | 去掉 fix 会失败，加回 fix 会通过 |
| requirements met | checklist 每条对应一条命令输出 |
| PR ready | tests pass + diff review 完成 |
| UI works | 在真实浏览器里跑完整流程（光跑测试不算） |

## 红旗清单

一旦主 agent 说出以下语言，立刻停止并要求补证据：

- "应该"
- "大概"
- "看起来"
- "理论上"
- 没有命令就说好
- "子 agent 说通过了"
- 依赖 > 5 分钟前的输出

## 流程

1. 接收声明（done / fixed / passing / ready 等）。
2. 对照「声明 vs 证据表」，确定需要什么证据。
3. 检查现有输出是否新鲜（≤ 5 分钟且文件未改动）；不新鲜则重新运行。
4. 现场执行命令，读完整输出和退出码。
5. 按输出模板汇报结果。
6. pass → 允许声明成立，继续 `ship`；fail → 停止，转 `diagnose`。

## 输出

```markdown
验证：
- 命令：`<cmd>`
- 时间：<现场跑的时间>
- 结果：pass | fail
- 证据：<短输出摘要 + 关键退出码>
```

## 停止条件

- **命令失败**：不报 done，不解释为什么"其实没问题"，转 `diagnose`。
- **无法找到可执行的验证命令**：明确说"未验证"，不假装。不猜，不估，不承诺。
- **子 agent 返回"通过"但无原始输出**：视为未验证，要求补跑。
