# Review

## 铁律

1. 先找真实风险，风格问题靠后。P0/P1 不解决前不放行。
2. 自审要按 PR 审同等标准执行：先读意图，再逐文件找可复现风险，不给自己开绿灯。
3. 没有具体 `file:line` 的 finding 不算 finding，不得写入报告。

## 何时不用我

- diff < 20 行且改的是测试或文档——直接看，无需走流程。
- 用户只想跑 lint / formatter——执行工具，不用我。
- 只是想理解代码意图，不是找 bug——用 `clarify` 或直接解释。

## 何时使用

- 用户说 "review"、"帮我看看"、"PR 审查"、"发布前 review"。
- 进入 `ship` 流程前的质量闸门。
- 自己刚写完代码要自审，防止带着理解盲点上线。

## 自审 vs PR 审

| 模式 | 触发 | 关键差异 |
|---|---|---|
| **自审** | 你刚写的代码 | 额外检查自己最容易跳过的边界、错误路径和测试缺口 |
| **PR 审** | 别人的代码 | 先读 plan/spec/issue 理解意图，再直接审 |

需要更深的方案级攻击时，转 `grill --redteam`，把 diff 或设计说明作为产物传入。

## 流程

1. **查看变更**：`git diff HEAD` 或 `gh pr diff <PR>`，列出所有变更文件。
2. **理解意图**：读 plan/spec/issue，明确这段代码要解决什么问题、预期行为是什么。
3. **自审模式**：先列出自己最可能漏看的 3 类风险（边界、错误路径、测试缺口），再逐项检查 diff。
4. **逐项检查**：
   - 正确性：逻辑、边界条件、类型断言。
   - 数据流：输入来源、变换、输出目标，有无数据被静默丢弃。
   - 错误处理：所有 error path 是否被捕获并正确传播。
   - 安全：输入校验、权限检查、敏感数据泄露风险。
   - 迁移风险：schema 变更、API 不兼容、依赖升级副作用。
   - 并发：锁、race condition、幂等性。
   - 高风险 diff 可读取 [`references/review-checklists.md`](references/review-checklists.md) 做扩展检查。
5. **测试覆盖**：变更的核心行为是否有对应测试；新分支/边界是否被覆盖。
6. **输出报告**：按 P0-P3 严重度排列 findings，每条带 `file:line`。
7. **需要复用时保存报告**：发布前或跨 session review 写到 `docs/reviews/<YYYY-MM-DD>-<slug>.md`；临时小 review 只在回复中输出。

## 优先级

| 级别 | 定义 |
|---|---|
| P0 | 数据丢失、安全破坏、生产事故 |
| P1 | 高概率 bug 或严重回归 |
| P2 | 边界 bug、重要测试缺失 |
| P3 | 可维护性 / 清晰度 |

## 输出

```markdown
## Findings
- [P1] <问题描述> — `<file:line>`

## Open Questions
- <需要作者澄清的疑问>

## Test Gaps
- <缺口描述，或 none>

## 产物位置
<保存路径；未保存则写"inline only">
```

无阻塞问题时：
```
未发现阻塞问题。剩余风险：<简述>
```

## 停止条件

- 没有 diff 或审查目标——先问清楚目标再继续。
- 不理解预期行为——先读 plan/issue，或向用户提问，不猜测。
- 需要更深的 adversarial 评估——转 `grill --redteam`。
- 发现 P0/P1——block 发布，回到 `diagnose` 修复或 `tdd` 补测试，修完再走 `ship`。
