# Implement

实现已经明确的完整任务。不要重新发散需求，不要顺手扩大范围。

## 输入

优先使用 `shape` 产出的实现说明；也可以使用用户给出的明确任务说明或具体修改要求。未直接给出实现说明时，默认从 `docs/implementation-specs/` 查找相关说明。

开始前确认：

- 目标和验收标准足够清楚。
- 本次范围完整且可执行。
- 没有必须先由用户决定的关键取舍。

如果以上不成立，停止实现并回到 `shape`。

## 流程

1. **读材料**：读取实现说明、`CONTEXT.md`、相关 ADR、现有代码和测试。
2. **找入口**：定位相关模块和改动区域，优先沿用仓库已有模式。
3. **拆执行计划**：把完整任务拆成可完成的步骤；复杂任务用计划跟踪。
4. **并行实现**：任务较大且边界清楚时，把独立子任务交给 subagent；主 agent 负责约束范围、整合结果和最终验证。
5. **实现全部内容**：覆盖实现说明中的所有任务和验收标准，避免无关重构和格式 churn。
6. **验证**：运行覆盖整体功能的测试、类型检查、构建或手动验证。
7. **收尾**：说明改了什么、怎么验证、还有什么风险。

## Subagent 规则

- 只把边界清楚、输入输出明确、互相依赖低的任务交给 subagent。
- 每个 subagent 只负责一个子任务或一类文件，避免多人同时改同一处。
- 给 subagent 的任务必须包含相关实现说明、范围、验收标准和禁止事项。
- 主 agent 必须 review subagent 结果，整合冲突，并运行最终验证。
- 不要把需求澄清、关键架构决策、最终验收外包给 subagent。

## 收尾

不要强制套固定模板。按实际情况说明：

- 完成了哪些内容。
- 修改了哪些关键区域。
- 运行了哪些验证，结果如何。
- 使用 subagent 时，说明拆给了谁、结果是否已整合。
- 还有哪些未完成事项、阻塞或风险；没有就不要写空栏目。

## 完成标准

- 实现说明或输入需求中的验收标准已全部处理；无法处理的项已明确说明原因。
- 所有本轮改动已整合到同一工作区，没有未检查的 subagent 结果。
- 相关代码、测试、配置、文档或迁移已按需要同步更新。
- 已运行与改动风险匹配的验证；未能验证的原因和影响已说清楚。
- 没有已知无关改动；残留风险具体、可行动。
