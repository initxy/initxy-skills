# OS

## 铁律

1. os 只路由，不执行——任何实质输出（代码、文档、决策）都必须由具体 skill 完成。
2. 说"完成/修复/通过"之前必须先走 `verify`；若用户明确跳过，必须标记 `未验证`，不能用完成语气。
3. 立项不清晰时，先用 `grill --align` 对齐关键假设；已有产物才用 `grill --redteam` 攻击。

## 何时不用我

- 任务明确且 30 分钟可逆——直接调对应 skill，不用经 os 分发。
- 单一 skill 已能覆盖全部意图——跳过 os。
- 低价值事务（改个变量名、查个 API）——GSD 逃生口，直接做。

## 何时使用

- 用户说"帮我完整走一遍"或不知道从哪里开始。
- 跨越多个阶段的复合任务（如"立项 + 设计 + 开发"）。
- 需要在阶段之间决定是否继续或分叉（如 refactor vs. 从头设计）。
- 排查 os 本身的路由错误或 skill 覆盖缺口。

## 路由表

```
阶段        用户意图                          → skill
------      --------------------------------  --------
控制面      初始化环境 / 安装 skill 包          → setup-os
            跨 session 传递上下文              → handoff

立项        要不要做这件事 / 值不值得           → assess
            想法混乱 / 需要被追问              → grill --align
            已有产物 / 需要找漏洞              → grill --redteam

定义        明确问题边界 / 写需求               → clarify
            在多个方向里做最终决定              → decide

设计        技术方案 / 模块划分 / 数据流        → design
            需要写正式 RFC / 多方评审           → rfc
            现有代码结构有问题 / 重构计划        → refactor

构建        拆任务 / 写执行步骤                 → plan
            TDD / 写测试 / 驱动实现             → tdd
            定位 bug / 分析错误                 → diagnose

发布        验收 / 功能门控 / 验证是否真的好了  → verify
            代码 review / PR 评审               → review
            发布 / 打包 / 上线                  → ship

沉淀        扫描知识 / 发现盲点                 → scan
            提炼心智模型 / 写学习笔记           → learn
            把结论写进项目记忆                  → capture
            把个人认知写回数字自我              → sediment
```

## 默认链路

```
用户意图 (模糊)
    │
    ▼
[assess] ── 不值得 ──→ STOP
    │
    ▼
[grill --align]? ── 假设太多 / 需求混乱时触发
    │
    ▼
[clarify] ──→ [decide]?   ← 有多个方向时
    │
    ▼
[design] ──→ [rfc]?       ← 需要正式评审时
    │         └─ [refactor]? ← 旧代码结构优先
    ▼
[plan] ──→ [tdd]
    │
    ▼
[verify] ◄── 闸门，未通过回到 [tdd] / [diagnose]
    │
    ▼
[review] ──→ [ship]
    │
    ▼
[learn] ──→ [sediment]? / [capture]?
```

## Bug 链路

```
bug 报告 / 异常
    │
    ▼
[diagnose] — 定位根因，输出假设
    │
    ▼
[tdd] — 写失败测试，锁定复现
    │
    ▼
[verify] — 证明修复有效（闸门）
    │
    ▼
[capture]? — 如果这个 bug 揭示了系统性问题，写进记忆
```

## 学习链路

```
学完一篇文章 / 完成一个项目
    │
    ▼
[scan]? — 主动扫描知识盲点（可选）
    │
    ▼
[learn] — 提炼心智模型 + 生成检验题
    │
    ├──→ [sediment] — 个人认知、原则、习惯 → 写回数字自我
    └──→ [capture]? — 项目经验 / 决策上下文 → 写回项目记忆
```

## 规则

1. 优先选最小匹配 skill——能用 `diagnose` 解决的不走完整流程。
2. 说"完成/修复/通过"之前必须先走 `verify`，不接受"应该好了"等弱证据。
3. 立项假设超过 3 个或需求有争议，优先走 `grill --align`；已有文档后再 `grill --redteam`。
4. 数字自我沉淀（个人原则/习惯/认知）只走 `sediment`，不走 `capture`。
5. 低价值、30 分钟内可逆的事不走全流程——GSD 逃生口，直接做。
6. 相邻 skill 之间传递产物，不重复收集输入；每个 skill 只看自己阶段。
7. 路由决策本身有争议时，用 `decide` 裁定，不由 os 自行猜测。

## 停止条件

- 已把用户意图映射到具体 skill 并移交——os 工作结束。
- 用户明确说"跳过规划直接做"——激活 GSD 逃生口，移交目标 skill，并在输出中标明跳过的风险。
- 发现意图无法映射到任何现有 skill——上报缺口，建议走 `setup-os` 或新建 skill。
