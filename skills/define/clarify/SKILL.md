# Clarify

## 铁律

1. 不规划模糊工作。先定义问题，再谈方案。
2. 区分问题和方案：用户说"我要做一个 App"是方案，不是问题；你的任务是挖出那个问题。
3. 验收必须可观察——不接受"用户体验更好"、"更快"、"更清晰"这类无法核查的描述。

## 何时不用我

- 用户已经清楚要做什么，只是要执行 — 直接进 `plan` 或 `tdd`，不用我。
- 用户已经有强立项假设并要求被攻击 — 走 `grill` 更合适。
- 是 A vs B 真 trade-off，选项已明确 — 走 `decide`。

## 何时使用

- "我想做 X，但还没想清楚"
- "帮我理一下这个需求"
- 缺目标、用户、范围、非目标、验收任意一项
- 用户直接说方案（"做个弹窗"），没说要解决什么问题
- 进入 `design` / `rfc` / `plan` 前，定义文档不存在或不完整

## 有效问题清单

在澄清过程中优先使用以下问题模板切开模糊（每次只问一个）：

1. "对谁来说这件事值得做？他们今天怎么解决这个问题？"
2. "三个月后，怎么知道这件事做对了？"
3. "如果只能砍掉一个功能，砍哪个？为什么？"
4. "如果完全不做，会发生什么？能接受吗？"
5. "已经有什么类似方案被试过但没成？原因是什么？"
6. "最怕这件事做完之后出现什么结果？"
7. "谁会反对这个定义？他们的理由是什么？"

## 流程

1. 读取用户输入，判断是否已有清晰问题定义。如果已有，跳到步骤 5 验证完整性。
2. 有歧义时，一次只问一个问题（从「有效问题清单」选最能切开当前模糊的那个）。
3. 区分问题和方案：如果用户描述的是方案，追问"这个方案要解决的根本问题是什么？"
4. 填充输出模板各项；至少强制写一个非目标（"不做"至少一条）。
5. 验收条件逐条检查可观察性：能写成"当 X 发生时，Y 指标达到 Z"才算合格。
6. 开放问题不超过 3 个；每个注明 owner（用户决策 / 团队决策 / 可后续调研）。
7. 需要复用时，保存定义文档到 `docs/clarify/<slug>.md`；告知用户路径。

## 输出

```markdown
## 问题
<一句话，说清楚谁有什么痛点，当前结果是什么>

## 目标
<可观察结果：当 X 发生时，Y 指标达到 Z>

## 用户 / 受众
<具体角色，而非"所有人">

## 范围
- 做：<明确边界>
- 不做：<至少一条非目标>

## 验收
- [ ] <可验证条件 1>
- [ ] <可验证条件 2>

## 开放问题
- <问题> — owner: <用户 / 团队 / 调研>
```

## 停止条件

- 开放问题超过 3 个 — 停止，先让用户决策，再继续。
- 没有至少一条可观察验收信号 — 不进入 `design` / `plan` / `rfc`。
- 已经是明确选项对比（A vs B）— 转 `decide`。
- 立项假设需要被攻击以验证 — 转 `grill`。
