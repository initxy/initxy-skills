# Decide

## 铁律

1. **条件影响结论时，不给无条件推荐。** 用「如果 X 选 A，如果 Y 选 B」的条件式格式，而不是「选 A」。
2. **先判断门类型，再决定决策严肃度。** 双向门（可逆）bias to action；一向门（难回滚）要强证据 + ADR。
3. **证据弱但影响大，必须标记。** 高影响决策遇到弱证据，标 `evidence: weak` 并建议先走 `grill --redteam`。

## 何时不用我

- **一个选项明显更好** — 没有真实 trade-off，不需要决策矩阵，直接推荐并解释原因。
- **选项本身还不清楚** — 转 `clarify`，先把问题定义清楚再决策。
- **用户只想被挑战，而非决策** — 转 `grill --redteam`，让怀疑论者攻击已有方向。

GSD 原则：30 分钟内可逆的选择，不值得走完整流程，直接给意见。

## 何时使用

- 「A 还是 B？」/ 「用哪个？」/ 「推荐一个」
- 技术选型（框架、数据库、云服务、工具链）
- 路线选择（架构方案、产品方向、执行策略）
- 供应商 / 外包 / 合作方评估
- 存在真实选项 + 真实 trade-off，需要结构化权衡

## Bezos 一向门 vs 双向门

借鉴自 Jeff Bezos 致股东信中的门类比。

| 属性 | 双向门（Two-Way Door）| 一向门（One-Way Door）|
|---|---|---|
| 可逆性 | 可以回头 | 极难或代价极高 |
| 典型例子 | 技术框架试用、UI 改版、A/B 测试 | 架构重写、数据迁移、关键合同、公开 API |
| 决策 bar | 低；信息够用即可行动 | 高；需要强证据 + 充分讨论 |
| 推荐行动 | Bias to action；事后调整 | 写 ADR；考虑配 `grill --redteam` 做对抗审查 |
| 错误代价 | 可控、可纠正 | 长期影响，难以撤销 |

**判断方法**：问「如果这个决定是错的，六个月后我们能以合理代价回滚吗？」能 → 双向门；不能 → 一向门。

## 流程

1. **定义决策问题，不定义实现。** 用一句话描述「我们要在什么约束下实现什么目标」，而不是「怎么做」。
2. **判断门类型。** 双向门继续；一向门提示写 ADR，建议配 `grill --redteam`。
3. **列出维度，最多 5 个。** 每个维度给权重（高/中/低）和衡量方式（可量化优先）。
4. **逐选项评估。** 每个选项给匹配度评分 + 证据来源 + 代价。
5. **证据优先级**：亲手验证 > 数据 > 案例 > 文档 > 观点。弱证据必须标注。
6. **写出放弃了什么。** 明确列出被排除选项的代价侧，不只说哪个好。
7. **给条件式推荐。** 若结论依赖条件，用「如果 X 选 A，如果 Y 选 B」格式。
8. **写复评触发条件。** 什么信号出现时应该重新评估。
9. **一向门决策：转 `capture` 写 ADR。** 归档决策上下文、备选方案、选择理由、放弃了什么；本 skill 不直接写文件。

## 停止条件

- **选项不清楚** → 转 `clarify`，先定义再决策。
- **没有真实 trade-off**（一个选项在所有维度占优）→ 跳出流程，直接给推荐 + 理由。
- **高影响 + 证据弱** → 标 `evidence: weak`，建议先走 `grill --redteam` 做对抗审查，再做决策。
- **一向门决策无 ADR** → 转 `capture` 写 ADR；若用户跳过，标记"未归档，未来上下文会丢失"。
