# Shape

把模糊想法收束成完整实现说明。**核心动作是追问，不是写文档**——实现说明是设计树走完、关键决策定了之后自然落下的副产物。设计没收敛就不要写实现说明。

## 流程

1. **读上下文**：先读 `CONTEXT.md`、`docs/adr/`、相关代码和现有任务说明。能从这些材料查到的，自己查，不要问用户。
2. **复述目标**：用一句话说清用户真正要达成什么；含糊就先追问，不要带着猜测往下走。
3. **追问到收敛**：见下方「追问」。这是 shape 的主要工作，多数时间花在这里。
4. **沉淀长期信息**：
   - 新领域术语、系统边界、稳定约定写入或建议写入 `CONTEXT.md`。
   - 影响后续实现路径、回滚成本较高、未来会被追问原因的取舍写入 ADR。
   - 小 UI 调整、简单 bug、一次性实现细节不写长期文档。
5. **收敛后再成稿**：只有当设计树走完、关键决策都定了，才把讨论结论整理成实现说明（含边界、任务拆分、依赖顺序、验收标准），详略由复杂度决定。

## 追问（核心）

像访谈一样把每个会影响实现的点问到底，直到和用户达成共同理解。

- **一次只问一个问题**，问完等回答再问下一个。一次抛一堆会让人懵，也没法顺着答案继续往下走。
- **沿设计树逐分支走**：先定前置决策，再问依赖它的下游决策，一个一个解依赖；不要跳着问彼此还没定的点。
- **每个问题都带上你的推荐答案和理由**。让用户确认或推翻你的判断，而不是逼他从零作答。
- **能查的自己查**：能从代码、文档、上下文判断的，去查，别拿来问用户。
- **只问要紧的**：只问会改变实现路径、边界或验收标准的问题；不影响结果的细节不问。
- **收敛判据**：关键决策点都已确定、没有阻塞实现的未决项，才算谈完，才进入成稿。

**例外**：用户明确说「只要个方案 / 别访谈 / 直接给建议」时，不强行逐题追问——直接给出推荐方案和理由，并标出你替他做的关键假设，让他事后纠正。

## 产物路径

- 小任务可以只在对话里输出实现说明。
- 复杂任务、跨 session 任务、需要交给 `implement` 或 `review` 复用的任务，写入文件。
- 默认写到 `docs/implementation-specs/<YYYY-MM-DD>-<slug>.md`。
- 文件名 slug 用英文；标题和正文用中文，技术术语按 `AGENTS.md` 的语言约定保留英文原词。

## 实现说明格式

```md
## Goal

## Non-goals

## Context

## Decisions

## Implementation plan

## Task breakdown

## Dependencies / sequencing

## Acceptance criteria

## Risks

## Files / areas to inspect
```

## 完成标准

- 追问已收敛：关键决策都定了，`Decisions` 里没有阻塞实现的未决项（这是成稿的前提，不满足就继续追问）。
- `Goal` 能一句话说清。
- `Non-goals` 排除了最容易蔓延的范围。
- `Implementation plan` 覆盖完整想法。
- `Task breakdown` 能看出哪些任务必须串行、哪些可以并行。
- `Acceptance criteria` 能被 `review` 用来判断是否达标。
- 需要跨 session 或后续复用时，已把实现说明写入约定路径。
- 长期有效的信息已写入或明确建议写入 `CONTEXT.md` / ADR。
