# Shape

把想法压成完整实现说明。不要急着实现；先让目标、边界、决策、任务拆分和验收标准清楚。

## 流程

1. **读上下文**：如果仓库里有 `CONTEXT.md`、`docs/adr/`、现有任务说明或相关代码，先读取足够材料。
2. **复述目标**：用一句话说明用户真正要达成什么；发现目标含糊时先追问。
3. **收边界**：明确本次做什么、不做什么、不能破坏什么。
4. **找决策点**：列出会影响实现路径的关键选择；每个选择只保留可行动的取舍。
5. **沉淀长期信息**：
   - 新领域术语、系统边界、稳定约定写入或建议写入 `CONTEXT.md`。
   - 影响后续实现路径、回滚成本较高、未来会追问原因的取舍写入 ADR。
   - 小 UI 调整、简单 bug、一次性实现细节不要写长期文档。
6. **拆任务面**：覆盖完整想法，把工作拆成可执行任务、依赖关系和推荐顺序；复杂任务标出可并行部分。
7. **输出实现说明**：形成足够支持完整实现的说明，详略由任务复杂度决定。

## 产物路径

- 小任务可以只在对话里输出实现说明。
- 复杂任务、跨 session 任务、需要交给 `implement` 或 `review` 复用的任务，写入文件。
- 默认写到 `docs/implementation-specs/<YYYY-MM-DD>-<slug>.md`。
- 文件名用短英文 slug；标题和正文可以用中文。

## 追问规则

- 一次最多问 3 个问题。
- 只问会改变实现路径、边界或验收标准的问题。
- 能从代码、文档或上下文判断的，不要问用户。
- 用户只想要建议时，给出推荐方案和理由；不要强行访谈。

## 实现说明格式

```md
## Goal

## Non-goals

## Context

## Decisions

## Implementation plan

## Task breakdown

## Dependencies / sequencing

## Acceptance criteria

## Risks

## Files / areas to inspect
```

## 完成标准

- `Goal` 能一句话说清。
- `Non-goals` 排除了最容易蔓延的范围。
- `Decisions` 中没有阻塞实现的未决项。
- `Implementation plan` 覆盖完整想法。
- `Task breakdown` 能看出哪些任务必须串行、哪些可以并行。
- `Acceptance criteria` 能被 `review` 用来判断是否达标。
- 需要跨 session 或后续复用时，已把实现说明写入约定路径。
- 长期有效的信息已写入或明确建议写入 `CONTEXT.md` / ADR。
