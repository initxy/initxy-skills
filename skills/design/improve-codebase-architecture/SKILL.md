# Improve Codebase Architecture

找架构摩擦，输出 deepening candidates。目标：更高局部性、更高杠杆、更窄接口、更好测试。

## 词汇

统一使用这些词：

| 词 | 含义 |
|---|---|
| 模块 | 有接口和实现的任意单位：函数、类、包、业务切片 |
| 接口 | 调用方必须知道的一切：签名、不变量、错误、顺序、副作用、配置 |
| 实现 | 模块内部代码 |
| 接缝 seam | 能替换行为而不原地改代码的位置 |
| adapter | 满足某个接口的具体实现 |
| 深模块 | 小接口承载大量行为 |
| 浅模块 | 接口复杂度接近实现复杂度 |
| 局部性 | 修改、bug、知识集中在一处 |
| 杠杆 | 一个接口服务 N 个调用方 / 测试 |

禁用模糊词替代：不要把 seam 写成边界，不要把接口只理解成类型签名。

## 铁律

1. 先找摩擦，再提方案；不做审美式重构。
2. 只产出候选报告，不直接改代码。
3. 每个候选必须过删除测试：删掉后复杂度是消失，还是扩散到调用方。
4. 候选最多 3 个；超过说明扫描边界太大。
5. 与 ADR 冲突必须标明，不静默推翻。

## 何时不用我

- 已有具体 bug / 失败 → 先 `diagnose`。
- 已有具体摩擦案例且用户要直接重构 → 走 `refactor`。
- 新系统设计 → 走 `design` / `rfc`。
- 用户只想理解代码 → 走 `zoom-out`。
- 单文件小整理 → 直接做，不需要架构体检。

## 何时使用

- 代码难测、难改、难导航，但还没锁定具体改法。
- 需要定期架构体检。
- 大功能前想先找结构风险。
- 重复逻辑、浅 wrapper、跨层泄漏、调用方知道太多。
- 想让 repo 更适合 agent 后续维护。

## 流程

1. **读上下文**
   - 读 `AGENTS.md` / `CONTEXT.md`。
   - 读相关 `docs/adr/`。
   - 不存在就跳过，不创建。

2. **限定扫描范围**
   - 有用户给的目录/feature：只扫该范围。
   - 没给范围：从最近高 churn、核心入口、测试薄弱区选 1 个切片。
   - 不全仓泛扫。

3. **找摩擦信号**
   - 理解一个概念要跳 5+ 文件。
   - 调用方必须知道实现细节或顺序约束。
   - 纯函数为测试被抽碎，但真实 bug 藏在编排里。
   - 一个 seam 只有一个 adapter，且没有真实变化点。
   - 相同规则散落 3+ 处。
   - 测试绕过 public interface。

4. **删除测试**
   - 删除后复杂度消失 → 浅模块，候选：删除/合并。
   - 删除后复杂度扩散 → 深模块，候选：保留并收窄接口。

5. **产出候选**
   - 1-3 个。
   - 每个候选必须有文件、问题、改法、删除测试、测试影响、强度。
   - 不写完整接口设计。

6. **需要可视化时写 HTML 报告**
   - 用户要求报告 / 候选关系复杂 / 模块超过 6 个时，读 [`references/html-report.md`](references/html-report.md)。
   - 报告写到 OS 临时目录：`${TMPDIR:-/tmp}/architecture-review-<timestamp>.html`。
   - 不写进 repo。

7. **等待用户选择**
   - 报告后问：`先深挖哪个？`
   - 用户选定后再走 `grill --align` 或 `refactor`。

## 输出

```markdown
## 扫描范围
- <目录 / feature / 入口>

## 候选方案

### 1. <名称>
- 文件：
- 问题：
- 改法：
- seam：
- 删除测试：
- 局部性收益：
- 杠杆收益：
- 测试影响：
- ADR 冲突：
- 强度：strong | worth exploring | speculative

### 2. <名称>
...

## 优先建议
<先做哪个，为什么。一段话。>

## 下一步
先深挖哪个？
```

## 停止条件

- 没有扫描范围且 repo 太大 → 要求用户给目录 / feature / 最近痛点。
- 候选全是审美问题 → 输出“不建议重构”。
- 方案已具体到可改代码 → 转 `refactor`。
- 触及重大技术决策 → 转 `decide` / `rfc`。
