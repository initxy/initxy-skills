# Zoom Out

把局部代码放回系统地图里。目标是减少迷路，不是总结文件内容。

## 铁律

1. 只解释结构，不改代码。
2. 用项目领域词汇命名模块；有 `AGENTS.md` / `CONTEXT.md` 先读。
3. 从入口、调用方、数据流、依赖方向讲，不逐文件复述。
4. 不确定就标 `未知`，不要补脑。

## 何时不用我

- 用户已经要求修 bug / 写代码 → 直接用 `diagnose` / `tdd` / 对应实现流程。
- 只是问某个函数语法 → 直接解释。
- 要正式设计新系统 → 走 `design` / `rfc`。
- 要主动找架构债 → 走 `improve-codebase-architecture`。

## 何时使用

- 用户不熟某块代码，需要先建立地图。
- 修改前要知道谁调用谁、数据怎么流。
- review 前要理解改动落在哪个系统区域。
- 接手陌生 repo、陌生目录、陌生 feature。

## 流程

1. 读项目上下文：`AGENTS.md`、`CONTEXT.md`、相关 `docs/adr/`，不存在则跳过。
2. 定位入口：路由、CLI、job、handler、public API、测试入口。
3. 找调用方：`rg` 符号、导入、路由注册、事件订阅。
4. 找被调用依赖：DB、外部服务、队列、文件、共享模块。
5. 画一条主数据流：输入 → 核心转换 → 副作用 → 输出。
6. 标出 seam：哪些接口是可替换/可测试的位置。
7. 标出风险点：隐藏耦合、命名不一致、跨层依赖、测试盲区。

```markdown
2. `<path>`：<原因>

```

## 停止条件

- 找不到入口 → 要求用户给目录、文件、符号或 feature 名。
- 发现用户其实要改代码 → 转对应实现 skill。
- 发现结构问题已经很明确且用户要优化 → 转 `improve-codebase-architecture` 或 `refactor`。
