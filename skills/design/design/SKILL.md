# Design

## 铁律

1. 技术形状没清楚前，不写代码。
2. 草图不写完整 NFR、容量模型和合规分析——那是 `rfc` 的活；但必须标出最关键的失败路径。

## 何时不用我

- 改动是单文件单函数级别——直接动手，不需要流程。
- 涉及生产 NFR、多人评审或长期演进（>3 个月）——走 `rfc`。
- 改的是已有结构而非新功能——走 `refactor`。

## 何时使用

- 新功能或新模块，实现形态还没锁定。
- 范围 1-3 个模块，几天内能写完。
- 处于私下探索阶段，未到正式评审。

## 流程

1. 从必要行为出发，不从偏好技术栈出发。
2. 技术栈只在影响约束时讨论，否则跳过。
3. 按清晰接缝拆模块（借鉴 Ousterhout：深模块优先，接口窄）。
4. 先定核心数据结构，再定 helper。
5. 写一条完整端到端数据流（输入 → 转换 → 读写 → 输出）。
6. 标出 trade-off 和未决项。
7. 模块和数据流稳定后交给 `plan`。
8. **完成后按风险选择**：仍有未清楚分支 → `grill --align`；已有草图且风险高 → `grill --redteam`。

## 升级到 rfc 的信号

以下任一触发，停止 design，走 `rfc`：

- 跨 3+ 模块改动。
- 涉及外部接口契约（第三方 / 跨服务）。
- 有 SLA、性能或容量目标。
- 有合规或安全约束。
- 需要多人 review 或团队共识。
- 维护周期超过 3 个月。

## 停止条件

- 不确定值不值得做——转 `assess`。
- 目标仍模糊——转 `clarify`。
- 多个选项未决——转 `decide`。
- 升级信号触发——转 `rfc`。
- 改的是存量结构——转 `refactor`。
