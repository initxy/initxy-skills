# Refactor

## 铁律

1. 没有具体摩擦案例（哪个文件:行、哪个 commit、哪个 bug），不重构。"看起来不够优雅"不是摩擦。
2. 只为提升局部性和杠杆做改动——减少调用方需要知道的、暴露可替换的接缝。不为美感。
3. 与现有 ADR 冲突时，先在输出里标明冲突，不静默推翻。需要改决策先走 `decide`。

## 何时不用我

- 是新建系统或全新模块 — 走 `design` 或 `rfc`。
- 没有具体摩擦，只是"看着不爽"或"想练手" — 不重构，直接关掉。
- 改动半径 > 你能在一周内完成 — 先走 `plan` 切片，每片当独立任务再来。

## 何时使用

- 修一个 bug 要动 4 个以上文件，且这些文件逻辑上本应无关。
- 写测试时发现无法注入假依赖、无法隔离副作用。
- 同一段逻辑以轻微变体重复出现 3 次以上。
- 准备加大改动（新功能、性能优化）前，需要先扩 seam 降风险。
- bug 事后分析发现"如果有这个接缝，这个 bug 本可隔离"。

## 词汇

| 术语 | 定义 |
|------|------|
| 模块 | 接口 + 实现 |
| 接口 | 调用方必须知道的一切（签名、副作用、异常、顺序依赖） |
| 接缝（seam） | 能替换行为而不改调用方的位置 |
| 深模块 | 小接口承载大量有价值行为；值得保留并加强 |
| 浅模块 | 接口复杂度接近实现复杂度；候选删除对象 |

## 删除测试（Ousterhout）

想象删掉这个模块：
- 复杂度**消失** → 浅模块，考虑删掉或合并。
- 复杂度**扩散到调用方** → 深模块，保留并加强其接口。

每个候选改动都要过一遍删除测试，结果写进输出。

## 流程

1. **读上下文**：读 `CONTEXT.md` 和相关 ADR，记录现有约定。
2. **找真摩擦**：要求用户给出具体文件:行 / commit / bug 链接；没有则停。
3. **跑删除测试**：对涉及模块逐一判断深浅。
4. **提出候选**：1-3 个改动，不超过 3 个；超过说明边界没想清楚，先 `clarify`。
5. **写候选详情**：每个候选必须填满下方模板的所有字段。
6. **等用户确认**：确认前不动任何代码，不写任何 diff。
7. **完成后按风险选择**：候选还模糊 → `grill --align`；候选已成形且影响大 → `grill --redteam`。

## 输出

```markdown
## 摩擦案例
- <具体文件:行 / commit hash / bug 链接，一行一条>

## 候选方案

### 1. <名称>
- 文件：
- 问题：
- 改法：
- 接缝：（新增或暴露哪个 seam）
- 删除测试：（复杂度消失 / 扩散）
- 收益：
- 风险：
- 测试影响：
- 强度：strong | worth exploring | speculative

### 2. <名称>
（同上）

## 优先建议
<先做哪个，为什么——一段话，给出明确推荐，不说"看情况">

## 与现有 ADR 的冲突
<如无冲突写"无"；如有，列出 ADR 编号和冲突点，并注明需先走 `decide`>
```

## 停止条件

- 用户给不出具体摩擦案例 → 停止，不重构。
- 候选方案与 ADR 冲突且用户不想先 `decide` → 停止，标明风险。
- 改动半径评估超过一周工作量 → 停止，转 `plan` 切片后再回来。
- 候选方案删除测试结果全部是"复杂度消失" → 说明是浅模块堆砌，建议直接删除而非改造。
