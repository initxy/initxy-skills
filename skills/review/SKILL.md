# Review

以审查为主，不继续实现新功能。先找问题，再给结论。

## 输入

- 当前 diff、PR、分支或指定提交范围。
- `shape` 实现说明、任务说明或用户原始需求；未直接给出时，默认查看 `docs/implementation-specs/`。
- `CONTEXT.md`、ADR、仓库测试和编码约定。

缺少需求来源时，先按代码质量和风险审查，并明确说明无法完整判断需求符合度。

## 流程

1. **确定范围**：说明正在审查哪个 diff / 分支 / 文件集合。
2. **对照需求**：逐条检查验收标准是否被实现，是否有范围蔓延。
3. **检查风险**：关注 bug、边界条件、数据迁移、兼容性、安全、性能、并发、错误处理。
4. **检查测试**：判断测试是否覆盖关键路径和失败路径。
5. **输出 findings**：按严重程度排序，只写可行动问题。
6. **给结论**：说明是否有阻塞问题；没有问题时明确说未发现阻塞问题。

## Findings 格式

每个问题使用：

```md
- [P1] 标题
  文件/位置：
  问题：
  影响：
  建议：
```

严重程度：

- `P0`：会造成数据丢失、严重安全问题、核心流程不可用。
- `P1`：主要功能错误、明显回归、合入前必须修。
- `P2`：边界问题、测试缺口、可维护性风险。
- `P3`：小问题，不阻塞。

## 输出格式

```md
## Findings

## Questions

## Test gaps

## Verdict
```

## 完成标准

- 已覆盖需求符合度和代码质量两条线。
- findings 都有明确影响和建议。
- 没有把风格偏好伪装成缺陷。
- 结论能支持“合入 / 修改后再看 / 需要重新 shape”的下一步判断。
