# 引导式中文写作

## 核心原则

好文档不是把要点填进模板,而是把一个问题想透之后,自然长出来的结构。

所以这个 skill 的动作是「引导用户想透 + 帮他落笔」,不是「替他生成一篇」。一句话喂进去吐出整篇,正是 AI 味的根源。

## 何时用

- 主打:有想法密度的技术/思考长文、博客、随笔。也能写设计文档、README、提案。
- 两个入口:
  - **从零写**:走下面完整三阶段。
  - **已有草稿**:跳过阶段一,直接进阶段三(去 AI 味 + 风格 + 配图),需要时再回头补结构。

## 工作方式

逐节推进,中等颗粒度。不一次代写整篇,也不做「每节抛一堆选项让用户勾选」的重流程。

### 阶段一 · 摸意图、第一性拆解、共定大纲

先问清楚,一次只问一个,等回答再问下一个:

- 写给谁?他们读完应该改变什么:知道什么、相信什么、会做什么?
- 手里有哪些真材料:数据、亲历、代码、踩过的坑、明确的观点?
- 有没有目标平台、篇幅、格式上的约束?

再做第一性拆解:把题目拆到不能再拆的基本事实和硬约束,从这些往上推,而不是先套一个现成结论或别人的框架。

共定大纲后,落成 `.md` 骨架,每个小节一个标题加占位,作为逐节填充的脚手架。

### 阶段二 · 逐节推进

从最不确定的小节先写,通常是核心论点;开头和概述留到最后。对每个小节:

1. 先问这节的核心要点和素材,用户可以简答或丢一段过来。
2. 起草这节,起草时就套用下面「风格」和「配图」的规则。
3. 用户提修改,用 Edit 局部更新文件,不重印全文。

### 阶段三 · 通读收尾

- 用下面「去 AI 味」的清单整篇扫一遍。
- 查风格是否前后统一。
- 查图够不够、贴不贴、标签是否具体。
- 收尾要轻:不写「总结 / 结语 / 写在最后」式套话。

## 风格

两条主线,主次分明。

### 第一性原理(主)

- 论证靠事实和约束推导,不靠类比、不靠「业界一般认为」「大家都知道」推进。
- 遇到现成结论先问「凭什么」,把它还原成可检验的前提。
- 和传统冲突时,从头推优先。诉诸先贤不能替代论证。

### 传统文化(淡淡点缀)

- 偶尔一句契合的诗词、古文、典故,只在它真能点睛时落,绝不为引而引。
- 宁少勿滥:整篇以现代白话为底,一两处足矣。堆古文就是失败。

### 语气

像一个已经想清楚的人,平静地讲给同行听。不端着,不煽情,不喊口号。

## 去 AI 味

中文 AI 味多来自「套模板 + 讲正确的废话」。写每一句都过这关。

高频要删或改的:

- **段末总结句**(最高频):「这说明…」「由此可见…」「到这里…」「综上…」。上一句说完就停,别重述。
- **对比句式**:「不是 X 而是 Y」。只说结论 Y。
- **开篇套话**:「随着…的发展」「在当今…时代」「近年来…受到越来越多关注」。直接开讲。
- **升华句**:把具体观察硬拔高到人生道理。删掉,或换成具体建议。
- **自问自答**:改成直接陈述。
- **空洞形容词预判**:「更优雅:…」「逻辑很清晰:…」。删掉形容词,直接说内容。
- **三连排比连发**:一个可以,背靠背就是模式化,换节奏。
- **黑话**:

| 不用 | 用 |
|---|---|
| 赋能 | 帮助 / 提供 |
| 抓手 | 关键措施 |
| 闭环 | 完整流程 |
| 沉淀 | 积累 / 形成 |
| 对齐 / 拉通 | 统一 / 打通 |
| 非常 / 极其 | 很 |
| 值得注意的是 | (直接说结论) |
| 例如 | 比如 |

标点与排版:简体中文标点;引号用直角引号「」;中文与英文或数字之间留一个空格(版本 2.0、用 Docker 跑);不用破折号 ——,改逗号或句号;代码、JSON 键名、URL、路径原样不动。

### 速查清单

整篇写完过一遍:

- [ ] 有没有段末总结句?删。
- [ ] 有没有「不是…而是…」?只留结论。
- [ ] 有没有开篇或章节间的套话过渡?删。
- [ ] 有没有升华句、空洞形容词预判?删。
- [ ] 有没有黑话?替换。
- [ ] 破折号、中英留白、直角引号对不对?
- [ ] 每个核心概念配图了吗?
- [ ] 传统文化点缀是不是过量了?宁少勿滥。
- [ ] 收尾有没有「总结 / 结语」套话?删。

### 正反例

> ❌ 众所周知,随着云原生的不断发展,Kubernetes 已成为容器编排的事实标准。值得注意的是,它并不是银弹,本文将从多个维度深入探讨其落地挑战。

> ✅ 我们用 Kubernetes 跑了三年生产。它解决了部署,却把复杂度搬到了别处。这篇记录我们踩的五个坑,和各自的止损办法。

## 配图

每个核心概念、流程、对比、结构,配一张图。能用图说清的,就别写一大段。默认用 mermaid。

思维模式对应图类型:

| 想表达 | 用 |
|---|---|
| 拆解 / 概念结构 | mermaid mindmap 或 graph |
| 流程 / 时序 | flowchart / sequenceDiagram |
| 分类 / 对比 | markdown 表格 或 quadrantChart |
| 因果 / 依赖 | 有向 graph |
| 第一性推导链 | 自顶向下 graph,从基本事实推到结论 |

约定:

- 图用真实标签,不用「模块 A / 模块 B」这种占位。
- 默认 mermaid 内联,可进 markdown、可版本管理、好改。
- 目标平台不渲染 mermaid(比如公众号),提示用户把图导出成图片再贴。
- 要做精致的演示级架构图,交给 `architecture-diagram` skill,这里不重造。

## 完成标准

也是 `review` 的验收口径:

- 第一性结构成立:核心论点是从事实和约束推出来的,不是套来的。
- 去 AI 味清单全过,没有残留 tells。
- 核心概念都有图,图贴切、标签具体。
- 风格淡而准:第一性为主,传统点缀不过量。
- 事实可核,数字、引用、代码真实可用。
- 收尾不套话。
