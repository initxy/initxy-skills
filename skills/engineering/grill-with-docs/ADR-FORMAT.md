# ADR 格式

ADR 放在 `docs/adr/`，编号递增：`0001-slug.md`、`0002-slug.md`。

懒创建 `docs/adr/`，只有第一个 ADR 需要时才建。

## 模板

```md
# {简短决策标题}

{1-3 句话：背景是什么，决定了什么，为什么。}
```

足够了。ADR 的价值是记录“做了这个决策”和“为什么”，不是填满章节。

## 可选章节

只有确实有价值时才加：

- `Status` frontmatter：`proposed | accepted | deprecated | superseded by ADR-NNNN`
- `Considered Options`：拒绝方案值得记住时使用。
- `Consequences`：有非显而易见后果时使用。

## 编号

扫描 `docs/adr/` 中最大编号，加一。

## 何时建议 ADR

必须同时满足：

1. 难逆转。
2. 没上下文会让未来读者意外。
3. 真有取舍。

容易逆转就不记；不意外就不记；没有替代方案也不记。

适合记录：

- 架构形状：monorepo、事件溯源写模型、Postgres 读模型等。
- 上下文之间的集成模式：事件、HTTP、队列等。
- 有锁定成本的技术选择：数据库、消息总线、认证、部署平台。
- 边界与范围：哪个上下文拥有 Customer 数据，其他上下文只引用 ID。
- 有意偏离常规的做法。
- 代码中看不出来的约束。
- 拒绝过且不显然的替代方案。
