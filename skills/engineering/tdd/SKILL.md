# Test-Driven Development

## 原则

测试应通过公开接口验证行为，而不是验证实现细节。代码可以彻底重构，测试不该因此坏掉。

好测试偏集成风格：走真实代码路径和公开 API，描述系统“做什么”，不是“怎么做”。例如“用户可用有效购物车结账”就是规格。

坏测试耦合实现：mock 内部协作者、测私有方法、绕过接口查数据库。预警信号：行为没变，重构后测试坏了。

示例见 [tests.md](tests.md)，mock 规则见 [mocking.md](mocking.md)。

## 反模式：水平切片

不要先写所有测试，再写所有实现。这会产生糟糕测试：

- 批量写出的测试验证想象中的行为。
- 容易测数据结构、函数签名，而不是用户可见行为。
- 行为坏了测试仍过，行为没变测试却坏。
- 过早承诺测试结构。

正确方式：垂直切片。一个测试 -> 一个实现 -> 重复。

```text
错误：
  RED:   test1, test2, test3
  GREEN: impl1, impl2, impl3

正确：
  RED->GREEN: test1->impl1
  RED->GREEN: test2->impl2
  RED->GREEN: test3->impl3
```

## 流程

### 1. 计划

探索代码时使用项目领域词汇表，让测试名和接口词汇与项目一致，并尊重相关 ADR。

写代码前确认：

- [ ] 公开接口需要怎样变化。
- [ ] 哪些行为最值得测试。
- [ ] 是否有 [deep modules](deep-modules.md) 机会。
- [ ] 接口是否 [易测试](interface-design.md)。
- [ ] 列出要测的行为，而不是实现步骤。
- [ ] 用户同意计划。

问：“公开接口应该长什么样？最重要的待测行为是什么？”

不能什么都测。优先关键路径和复杂逻辑。

### 2. Tracer bullet

先写一个测试，确认系统的一件事：

```text
RED:   写第一个行为测试 -> 失败
GREEN: 写最少代码让它通过 -> 通过
```

### 3. 增量循环

剩余行为逐个处理：

```text
RED:   写下一个测试 -> 失败
GREEN: 最少代码通过 -> 通过
```

规则：

- 一次一个测试。
- 只写当前测试所需代码。
- 不提前实现未来需求。
- 测试只关注可观察行为。

### 4. 重构

全部 green 后看 [refactor candidates](refactoring.md)：

- [ ] 提取重复。
- [ ] 加深模块，把复杂度藏在简单接口后。
- [ ] 自然适用时使用 SOLID。
- [ ] 观察新代码暴露出的旧代码问题。
- [ ] 每个重构步骤后跑测试。

RED 状态绝不重构。

## 每轮检查

```text
[ ] 测试描述行为，不描述实现
[ ] 只使用公开接口
[ ] 内部重构后测试应继续有效
[ ] 代码只够当前测试通过
[ ] 没有 speculative feature
```
