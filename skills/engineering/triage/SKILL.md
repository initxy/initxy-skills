# Triage

把 issue tracker 上的 issue 移动到小型状态机中。

triage 期间发布到 issue tracker 的每条评论或 issue 都必须以下方免责声明开头：

```markdown
> *This was generated by AI during triage.*
```

## 参考文档

- [AGENT-BRIEF.md](AGENT-BRIEF.md)：如何写持久 agent brief。
- [OUT-OF-SCOPE.md](OUT-OF-SCOPE.md)：`.out-of-scope/` 知识库规则。

## 角色

两个 category：

- `bug`：某物坏了。
- `enhancement`：新功能或改进。

五个 state：

- `needs-triage`：maintainer 需要评估。
- `needs-info`：等待报告人补充信息。
- `ready-for-agent`：规格完整，可交给 AFK agent。
- `ready-for-human`：需要人类实现。
- `wontfix`：不会处理。

每个已 triage issue 应恰好有一个 category 和一个 state。状态冲突时先标出并询问 maintainer。

这些是标准角色名；真实标签可能不同。映射应由 `/setup-matt-pocock-skills` 提供。

状态流：未标记 -> `needs-triage`；再到 `needs-info`、`ready-for-agent`、`ready-for-human` 或 `wontfix`。`needs-info` 在报告人回复后回到 `needs-triage`。maintainer 可随时覆盖。

## 调用

maintainer 用自然语言调用 `/triage`：

- “Show me anything that needs my attention”
- “Let's look at #42”
- “Move #42 to ready-for-agent”
- “What's ready for agents to pick up?”

## 展示需要关注的 issue

查询 issue tracker，按 oldest first 分三桶：

1. Unlabeled：从未 triage。
2. `needs-triage`：待评估。
3. `needs-info` 且报告人在上次 triage notes 后有回复：需要重新评估。

展示数量和每个 issue 一行摘要，让 maintainer 选择。

## Triage 单个 issue

1. 收集上下文：读取完整 issue（正文、评论、标签、报告人、日期）。解析之前的 triage notes，避免重复提问。用项目领域词汇探索代码，并尊重相关 ADR。读取 `.out-of-scope/*.md`，找类似拒绝记录。
2. 推荐：给出 category 和 state 推荐、理由，以及相关代码库摘要。等待方向。
3. 复现（仅 bug）：先尝试复现，读取步骤，追踪代码，运行测试或命令。报告结果：复现成功、复现失败、信息不足。
4. 追问（需要时）：运行 `/grill-with-docs`。
5. 应用结果：
   - `ready-for-agent`：发布 agent brief 评论。
   - `ready-for-human`：同样结构，但说明为什么不能委托给 agent。
   - `needs-info`：发布 triage notes。
   - `wontfix` bug：礼貌解释并关闭。
   - `wontfix` enhancement：写入 `.out-of-scope/`，评论中链接它并关闭。
   - `needs-triage`：加角色；已有部分进展可评论。

## 快速状态覆盖

如果 maintainer 说“move #42 to ready-for-agent”，信任并直接改。先确认将执行的角色变化、评论、关闭动作，再做。跳过追问。若未写 grilling session 就转 `ready-for-agent`，询问是否要写 agent brief。

## Needs-info 模板

```markdown
## Triage Notes

**What we've established so far:**

- point 1
- point 2

**What we still need from you (@reporter):**

- question 1
- question 2
```

把已解决内容写进 established，问题要具体可执行。

## 恢复旧会话

读取之前 triage notes，检查报告人是否回答了未解决问题，先展示更新后的情况。不要重复问已解决问题。
