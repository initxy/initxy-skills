# Prototype

原型是“回答一个问题的一次性代码”。问题决定形状。

## 选择分支

先判断要回答什么：

- “这个逻辑 / 状态模型对吗？” -> [LOGIC.md](LOGIC.md)。做小型交互式终端 app，让用户推动状态变化。
- “这个应该长什么样？” -> [UI.md](UI.md)。在一个路由上生成多个明显不同 UI 变体，用 URL 参数和底部浮动条切换。

两类产物完全不同。判断错会浪费原型。如果含糊且用户不在线，按周边代码推断：后端 module 倾向 logic，页面 / component 倾向 UI，并把假设写在原型顶部。

## 通用规则

1. 从第一天起就是一次性代码，并明确标记。代码放在将来真实代码附近，但命名要让读者知道它是 prototype。
2. 一个命令可运行：`pnpm <name>`、`python <path>`、`bun <path>` 等，沿用项目 task runner。
3. 默认无持久化。状态在内存。若问题就是持久化，使用 scratch DB 或明确可删除的本地文件。
4. 不打磨：无测试，最少错误处理，不抽象。目标是快速学习然后删除。
5. 暴露状态：每次动作或变体切换后显示完整相关状态。
6. 完成后删除或吸收。原型回答问题后，要么删掉，要么把有效决策折进真实代码。

## 完成时

唯一值得保留的是答案。把“问题”和“答案”记录在 commit message、ADR、issue，或原型旁边的 `NOTES.md`。用户在线就快速确认；不在线就留占位，后续删除前补 verdict。
