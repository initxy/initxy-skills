# Logic Prototype

一个小型交互式终端 app，让用户手动驱动状态模型。适用于业务逻辑、状态转换、数据 shape：纸面上看起来合理，实际跑边界 case 才发现问题。

## 适用场景

- “这个状态机能处理 X 后接 Y 吗？”
- “这个数据模型能表达某种边界情况吗？”
- “写真实实现前，想感受 API 形状。”
- 用户需要按键并观察状态变化。

如果问题是“页面该长什么样”，用 [UI.md](UI.md)。

## 流程

### 1. 写清问题

写代码前，用一段话说明要验证的状态模型和问题。放 README 或文件顶部注释。原型回答错问题就是浪费。

### 2. 选语言

使用宿主项目语言。无明显 runtime 时问用户。不要为了原型引入新包管理器或 runtime。

### 3. 把逻辑隔离成可移植 module

真正回答问题的逻辑放在小而纯的 interface 后，之后可搬进真实代码。TUI 是一次性的，逻辑 module 不该是。

常见形状：

- 纯 reducer：`(state, action) => state`。
- 状态机：显式状态与转移。
- 一组处理普通数据类型的纯函数。
- 拥有清晰方法面的 class / module。

选择最贴合问题的形状，不要为了 TUI 方便而扭曲。保持纯净：无 I/O、无终端代码、无 `console.log` 控制流。

### 4. 构建最小 TUI

每次 tick 清屏并重绘整帧，用户看到稳定视图，而不是不断增长的 scrollback。

每帧两部分：

1. 当前状态：pretty print，diff 友好；字段名或标题可用 ANSI bold，次要信息 dim。
2. 快捷键：`[a] add user  [d] delete user  [t] tick clock  [q] quit`。

行为：

1. 初始化单个内存 state。
2. 每次读一个按键或一行。
3. dispatch 后重绘整帧。
4. 循环直到 quit。

整帧应装进一个屏幕。

### 5. 一个命令运行

把命令加入现有 task runner。没有 task runner 时，把命令写在原型 README 顶部。

### 6. 交给用户

给运行命令。用户会自己操作；他们说“等等，这不该可能”或“我以为会是 X”时，就是原型捕获到想法里的 bug。

### 7. 捕获答案

原型完成后，只保留答案。用户在线就问它教会了什么；不在线就在旁边留 `NOTES.md`。

## 反模式

- 不加测试。需要测试就不再是原型。
- 不连真实数据库，除非问题就是持久化。
- 不泛化，不考虑未来可能支持什么。
- 不把逻辑和 TUI 混在一起。
- 不把 TUI shell 发进生产。
