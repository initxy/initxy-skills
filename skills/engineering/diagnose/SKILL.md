# Diagnose

用于难查 bug。除非有明确理由，不跳阶段。

探索代码时，先读项目领域词汇表和相关 ADR，建立准确模型。

## 阶段 1：建立反馈环

这是本 skill 的核心。只要有快速、确定、agent 可运行的通过/失败信号，二分、假设测试、插桩都会有效；没有反馈环，就不要靠读代码猜。

优先顺序：

1. 在能触达 bug 的接缝写失败测试：单元、集成、e2e 都可。
2. 对运行中的 dev server 写 `curl` / HTTP 脚本。
3. 用 fixture 调 CLI，把 stdout 与已知正确快照做 diff。
4. 用 Playwright / Puppeteer 驱动 UI，断言 DOM、console、network。
5. 保存真实请求、payload、事件日志并离线 replay。
6. 写一次性 harness：只启动必要服务或 mock 依赖，用一个函数调用触发路径。
7. 对“偶发错误输出”跑 property / fuzz 循环。
8. 对版本、数据、依赖之间的回归写 `git bisect run` 可用 harness。
9. 用同一输入跑旧版与新版，做差分。
10. 最后才用 HITL 脚本；参考 `scripts/hitl-loop.template.sh`。

拿到反馈环后继续打磨：

- 更快：缓存 setup、缩小测试范围。
- 更准：断言具体症状，而不是“没崩”。
- 更稳：固定时间、随机种子、文件系统、网络。

非确定性 bug 的目标是提高复现率。循环 100 次、并行、加压、插入 sleep、缩窄时序窗口；1% 很难查，50% 就能查。

如果确实无法建立反馈环，停止并说明试过什么，向用户要可复现环境、HAR / 日志 / core dump / 带时间戳录屏，或请求加临时生产插桩。没有反馈环不要进阶段 2。

## 阶段 2：复现

运行反馈环，看 bug 出现。

确认：

- [ ] 失败模式就是用户描述的 bug。
- [ ] 多次运行可复现；偶发 bug 也要达到可调试频率。
- [ ] 已捕获精确症状：错误信息、错误输出、耗时等。

未复现不要继续。

## 阶段 3：提出假设

先列 3-5 个排序假设，再测试。每个假设必须可证伪：

> 如果 `<X>` 是原因，那么改动 `<Y>` 会让 bug 消失，或改动 `<Z>` 会让 bug 更严重。

说不出预测就是直觉，丢掉或收紧。

测试前把排序列表给用户看。用户可能知道刚发布了哪个变更，或哪些已排除；用户不在线时按你的排序继续。

## 阶段 4：插桩

每个探针必须对应阶段 3 的一个预测。一次只改一个变量。

工具优先级：

1. 调试器 / REPL；一个断点胜过十条日志。
2. 只在区分假设的边界打日志。
3. 不要“全打日志再 grep”。

所有调试日志加唯一前缀，如 `[DEBUG-a4f2]`，收尾时一条 grep 清干净。

性能分支：先基线测量，再二分。用 timing harness、`performance.now()`、profiler、query plan；先测量，后修复。

## 阶段 5：修复与回归测试

修复前先写回归测试，但前提是存在正确接缝。

正确接缝 = 测试按真实调用方式覆盖真实 bug 模式。太浅的测试会给假信心。

如果没有正确接缝，这本身就是发现：架构挡住了 bug 锁定。记录下来，后续交给 `improve-codebase-architecture`。

如果有正确接缝：

1. 把最小复现变成失败测试。
2. 确认失败。
3. 修复。
4. 确认通过。
5. 用阶段 1 的原始场景再跑一遍。

## 阶段 6：清理与复盘

完成前必须：

- [ ] 原始复现不再失败。
- [ ] 回归测试通过；没有接缝则已记录。
- [ ] 删除所有 `[DEBUG-...]` 插桩。
- [ ] 删除一次性原型，或移到明确 debug 目录。
- [ ] 在 commit / PR 信息里写清楚最终正确的假设。

最后问：什么能预防这个 bug？如果答案是架构问题，比如无测试接缝、调用方缠绕、隐藏耦合，带具体发现交给 `improve-codebase-architecture`。
