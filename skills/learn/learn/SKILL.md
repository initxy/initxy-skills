# Learn

## 铁律

1. 解释是为了能用，不为堆知识——每个概念必须落地到"你能拿它做什么"。
2. 真懂的标志是能答检验题，不是听过——不能出检验题说明还没搞清楚。
3. 学完必须做唯一性判定——不是所有内容都值得 sediment，默认不沉淀。

## 何时不用我

- 用户只要快速答案 — 直接答，不展开，不走模板。
- 内容是用户已熟悉的领域 — 直接回顾，不用我。
- 任务是过滤大量外部信息 — 转 `scan`。

## 何时使用

- "解释 X"
- "读一下这个"
- "帮我理解这段代码 / 论文 / 文章"
- "我想学 X"
- "这个概念是什么意思"

## 流程

1. **重建结构**：不照搬原文目录；按"核心机制 → 部件 → 边界"重组。
2. **用已知定义未知**（Feynman）：一句话定义必须引用用户已知的概念，不造新术语。
3. **给最小例子**：优先一个能运行/能复现的最小示例，少写长段解释。
4. **读代码库时先找入口**：main / handler / entry point，顺着控制流往下，不从文档开始。
5. **出 3 道检验题**：按「检验题设计」原则，确保题目要求推理而非背诵。
6. **给练习路径**：今天能做的最小练习 + 本周扩展各一条。
7. **强制唯一性判定**：学完必须回答判定框架，决定是否转 `sediment`。
8. **项目特有理解** → `capture`；**个人跨项目研究** → `sediment`。

## 检验题设计

好检验题：回答需要推理，不能照抄原文。

| 类型 | 示例 |
|------|------|
| 应用题 | "如果 X 发生，系统会怎样？" |
| 反例题 | "什么情况下这个方法会失败？" |
| 对比题 | "X 和 Y 在 Z 场景下有什么不同？" |

坏检验题：背诵性、术语对照（"X 的定义是什么？"）。

```markdown
- 今天能做的最小练习：
- 本周扩展：

## 唯一性判定（关键）
是否值得 sediment？回答以下：
- LLM 能直接答出我刚写的这些吗？ yes / no
- 这里面有我自己的判断 / 反例 / anti-pattern 吗？ yes / no
- 如果 LLM 能答 + 没有我的独特视角 → 不 sediment

```

## 停止条件

- 材料太大（> 一个章节/模块）— 先问目标问题或指定章节，再开始。
- 用户要行动方案 — 转 `plan`。
- 用户要选型 — 转 `decide`。
- 唯一性判定通过（LLM 答不出 + 有独特视角）— 转 `sediment`。
- 是过滤大量外部信息的需求 — 转 `scan`。
