# Sediment

## 铁律

1. **LLM 能直接答出来的，不存**——存了也只是知识抄写员，不是数字自我。
2. **网上随手能搜到的，不存**——搜索比知识库更新鲜。
3. 每一条必须能回答「为什么是我才能写出这个」——答不上来则不写。

## 何时不用我

- 是项目内知识（ADR、SOP、诊断经验）→ 转 `capture`（写进 repo docs/）。
- 是通用教程知识（框架文档、语言特性、概念解释）→ 不存；哪怕用户说"记一下"，也要先反问「你自己有什么独特判断或反例吗」。
- 是一次性事务（今天待办、本周计划、临时笔记）→ 不存。
- 唯一性判定 4 问中任何一问失败 → 不存，直接告知用户。

## 何时使用

- 用户形成了一个独特判断："我做 X 类项目默认选 Y，因为我吃过 Z 的亏"。
- 用户踩了反直觉的坑："教科书说 A，但在 B 场景下 A 让我栽了"。
- 用户发现跨领域联系："A 领域的 X 原理和 B 领域的 Y 完全是同一套机制"。
- "以后默认 X"——跨项目原则，不只针对当前项目。
- "我又一次发现 Y"——pattern 第二次出现，可能值得沉淀。
- `learn` 流程末尾「唯一性判定」通过，转到这里。

## 唯一性判定（写之前必过，4 问）

在写任何条目之前，逐一回答：

| # | 问题 | 失败条件 |
|---|---|---|
| Q1 | 这条内容 LLM 直接问能答出来吗？ | yes → 不存 |
| Q2 | 这是基于我自己的实践 / 反例 / 失败案例吗？ | no → 大概率不该存 |
| Q3 | 删掉这条 6 个月后我能在 LLM / Google 重新得到它吗？ | yes → 不存 |
| Q4 | 这条对应的是一个「我特别的判断 / 偏好 / 反模式」吗？ | no → 不存 |

4 问全部通过才继续。

## 四类应存内容

| 类型 | 特征 | 示例 |
|---|---|---|
| **个人判断** | 基于经验的非通用选择 | "我做数据密集型项目默认选 Postgres，因为我用 MongoDB 被 schema drift 坑过三次" |
| **个人反例** | 教科书说 A，但在 B 场景下 A 失败 | "Retry with exponential backoff 在幂等性未验证的场景下会制造重复数据，我在 X 项目踩过" |
| **个人 anti-pattern** | 自己重复踩的坑 | "我倾向于过早抽象 hook，导致后来 refactor 成本比直接写高" |
| **独到关联** | A 领域和 B 领域的非显然联系 | "Conway's Law 和 Goodhart's Law 在组织设计上是同一个机制的两面" |

## 目标位置

| 内容类型 | 目标路径 |
|---|---|
| 个人原则 / 跨项目选择 | `~/notes/principles/<slug>.md` |
| 跨项目研究 / 深度笔记 | `~/notes/research/<slug>.md` |
| 个人 anti-pattern | `~/notes/anti-patterns/<slug>.md` |
| 独到关联 / 跨域洞察 | `~/notes/insights/<slug>.md` |

规则：每条选**一个**位置，不分散，不重复。

## sediment vs capture

| 维度 | sediment（本 skill） | capture |
|---|---|---|
| 写到哪 | `~/notes/`（个人空间）| 项目 repo 内 `docs/` |
| 生命周期 | 跨项目，永久 | 与代码共生，随项目归档 |
| 读者 | 未来的自己 | 新人 / 新 agent 接手项目 |
| 独特性要求 | 必须——LLM 答不出 | 必须——项目特有，非通用知识 |
| 典型内容 | 原则、反例、anti-pattern、跨域洞察 | ADR、SOP、诊断经验、项目约定 |

同一条知识若 LLM 能直接答且非项目特有 → 既不 sediment 也不 capture，直接不写。

## 流程

1. **跑唯一性判定 4 问**——任何一问失败，停止，告知用户不写理由。
2. **选目标位置**——按「目标位置」表选唯一路径。
3. **写最小条目**——带 frontmatter，bullet 为主，不写长段叙事。
4. **强制找反例**——"这条在什么情况下不成立？"；没反例不算成熟原则，先问用户。
5. **链接相关条目**——查 `~/notes/` 中是否有同主题文件，追加引用。
6. **评估回写**——反复出现的原则考虑升级到 `~/.claude/CLAUDE.md` 全局约定（见元递归）。

## 输出模板

每条 sediment 文件的最小结构：

```markdown
---
title: <一句话>
why-unique: <为什么 LLM 答不出来 / 为什么是我才有>
triggered-by: <什么场景下这条该被想起>
evidence: <具体案例 / 反例 / commit / 项目链接>
---

<主体内容，简短，bullet 为主>

## 反例 / 边界
<什么时候这条不成立>
```

## 元递归

sediment 可以触发更高层级的回写（everything claude code 思路）：

- 反复出现的原则（3 次以上）→ 升级为 `~/.claude/CLAUDE.md` 全局约定，让所有 agent 默认遵守。
- 发现某个 skill 的默认栈有问题 → 回写 `rfc` 或 `design` skill 里的"默认栈"小节。
- 某类 anti-pattern 集中出现 → 在 `~/notes/anti-patterns/` 写总结条目，再回写 `tdd` / `verify` 的停止条件。

回写规则：只改已有文件，不新增 skill；改动必须能通过「铁律」检验。

## 停止条件

- 唯一性判定不过 → 不写；若是项目向，转 `capture`；若是通用知识，直接丢。
- 没有反例 → 先向用户要反例，再决定是否存。
- 是项目向知识 → 转 `capture`。
- 内容是一次性事务 → 不存，告知原因。
